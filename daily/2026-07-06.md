# setTimeout() 함수가 반환하는 값이 id였던 점에 대하여

## 오늘 배운 것

오늘은 라이브코딩으로 `Promise.myApply()`와 `debounce`를 구현하는 시간을 가졌다.

`myApply()`는 어제 `myBind()`를 구현하면서 `call()`과 `apply()`의 차이를 공부했기 때문에 어렵지 않게 구현할 수 있었다. 그리고 `debounce`도 단골 문제라서 어렵지않게 구현할 수 있겠지? 하고 시작했는데, 구현 도중 "`setTimeout()`이 끝나면 `timerId`를 초기화해야 한다."라는 힌트를 듣고 뇌가 멈췄다...

내가 작성한 코드는 아래와 같았다.

```js
function debounce(fn, delay) {
  let timerId;
  return function (...args) {
    clearTimeout(timerId);
    timerId = setTimeout(() => {
      fn.apply(this, args);
    }, delay);
  };
}
```

`debounce`가 호출될 때마다 새로운 `setTimeout()`이 등록되므로 굳이 `timerId`를 `null`로 초기화할 필요가 없다고 생각했다. 또한 `clearTimeout()` 역시 이전 타이머가 있을 때만 호출하면 된다고 생각했기에 위처럼 작성했다.

하지만 매번 `clearTimeout()`을 호출하면 `timerId`가 `undefined`일 때도 `clearTimeout()`이 호출되므로, `timerId`가 존재할 때만 `clearTimeout()`을 호출하도록 수정해야 한다는 힌트를 받았다.

그래서 아래와 같이 수정했다.

```js
if (timerId) clearTimeout(timerId);
```

그리고 내가 가장 크게 착각하고 있던 부분을 깨닫게 되었다.

### 내가 착각했던 점

"`setTimeout()`이 끝나면 `timerId`를 초기화해야 한다."를 아래와 같이 진행하면 된다고 생각했다.

```js
timerId = setTimeout(() => {
  fn.apply(this, args);
}, delay);
timerId = null;
```

1. `setTimeout()` 실행
2. `timerId`에 반환값 저장
3. `timerId = null`

의 실행순서를 거친다고 생각했기에 위와 같이 작성했으나 실제로는 그렇지 않았다.

### 실제 실행 과정

`setTimeout()`은 delay 동안 코드를 멈추는 함수가 아니다.

`setTimeout()`은 먼저 타이머를 등록한 뒤, 취소할 수 있도록 타이머의 식별자(ID)를 즉시 반환한다.

```js
timerId = setTimeout(() => {
  fn.apply(this, args);
  timerId = null;
}, 300);
```

위 코드의 실행 순서는 다음과 같다.

1. `setTimeout()`이 타이머를 등록한다.
2. 타이머의 식별자(ID)가 `timerId`에 저장된다.
3. 함수는 바로 종료된다.
4. 300ms가 지난 뒤 콜백 함수가 실행된다.
5. `fn.apply()`가 실행된다.
6. 마지막으로 `timerId = null`이 실행된다.

즉, `timerId`는 300ms 동안 계속 타이머의 식별자를 가지고 있다가, 콜백이 실제로 실행되는 시점에서야 `null`이 된다. 내가 생각했던 것처럼 "`setTimeout()`이 끝나면 바로 `null`이 되는 것"이 아니었던 것이다.

이번에 알게 된 사실은 `setTimeout()`이 반환하는 값이 타이머를 취소하기 위한 식별자라는 점이었다. 그래서 `clearTimeout(timerId)`가 가능한 것이고, 타이머가 실행 중인지 여부도 `timerId`를 통해 판단할 수 있다.

> 참고로 브라우저에서는 `setTimeout()`이 숫자 형태의 ID를 반환하고, Node.js에서는 Timeout 객체를 반환한다. 하지만 두 환경 모두 `clearTimeout()`에 전달할 수 있는 타이머 식별자를 반환한다는 점은 동일하다.

출제자분께서 계속 힌트 다 드렸다며 하시면 된다고 나를 격려하셨으나... 도무지 감이 오지 않아 같은 실수를 반복하며 "이거 아니에요?" 무새가 되었고, 결국 숨이 넘어가실 정도로 웃으셨다...😂

덕분에 `setTimeout()`의 동작 방식과 실행 순서를 확실하게 이해할 수 있었다.

### 왜 `timerId`를 초기화해야 하는가? 🤔

실제로 내가 작성한 코드도 동작하긴 한다. 하지만 `timerId`를 `null`로 초기화하는 이유는 현재 상태를 명확하게 표현하기 위해서이다. `timerId`가 `null`이면 현재 타이머가 등록되어 있지 않다는 것을 의미하고, `null`이 아니면 타이머가 등록되어 있다는 것을 의미한다.

따라서 `timerId`를 초기화하는 것은 코드의 가독성을 높이고, 상태를 명확하게 표현하는 좋은 습관이다.
