# 라이브코딩 스터디와 입사를 준비하며

오늘은 JavaScript 구현 문제를 정리하고 Shopify에 대해 공부했다. 그리고 TIL 사이트의 모바일 사이드바 UX도 개선했다.

## 오늘 배운 것

### `Array.prototype.flat()` 직접 구현하기

[../live-coding/array-my-flat.md](../live-coding/array-my-flat.md)

라이브 코딩 스터디에서 사용할 문제로 `Array.prototype.flat()`과 동일하게 동작하는 `myFlat()`을 구현했다.

처음에는 중첩된 배열을 재귀적으로 펼치기만 하면 될 것 같았지만, 실제 `flat()`의 동작에 가깝게 구현하려면 생각해야 할 조건이 꽤 많았다. 특히 빈 슬롯을 가진 희소 배열은 `undefined`가 아니라 해당 인덱스 자체가 존재하지 않는 것으로 취급한다는 점이 인상 깊었다. `flat()` 역시 이러한 빈 슬롯은 결과에 포함하지 않고 건너뛴다.

### `Map`으로 LRU Cache 구현하기

[../live-coding/lru-cache.md](../live-coding/lru-cache.md)

두 번째로는 LRU Cache를 구현했다.

예전에 프로젝트에서 Copilot이 비슷한 캐시를 구현해준 적이 있었지만, 당시에는 왜 그렇게 구현하는지 제대로 이해하지 못했다. 이번에는 직접 구현하면서 `Map`을 사용하는 이유와 캐시의 갱신 방식을 정리했다.

해당 캐시를 구현하기 위해서 `Map`을 사용하는 게 적합했는데, 그 이유는 `Map`이 삽입 순서를 유지하기 때문이다. 캐시에서 가장 오래 사용하지 않은 데이터를 제거해야 하는데, `Map`을 사용하면 가장 오래된 데이터가 항상 맨 앞에 위치하게 되기에 구현 과정이 다소 간단했다.

### Shopify의 구조 이해하기

[../shopify/01-shopify-overview.md](../shopify/01-shopify-overview.md)

Shopify 학습 로드맵의 첫 번째 단계로 Shopify가 어떤 플랫폼인지 정리했다. 카페24와 NHN Shopby와 같은 국내 쇼핑몰 솔루션과 비교하며 Shopify의 구조를 이해하려고 했다.

처음에는 Shopify 프론트엔드 개발이라면 단순히 화면을 구현하고 기능을 연결하는 일이라고 생각했다. 하지만 실제로는 Theme 개발, App 개발, Admin 확장 등 여러 영역으로 나뉘며, 각각 사용하는 기술과 접근 방식도 달랐다.

앞으로 Shopify를 공부할 때도 먼저 현재 작업이 **Theme을 수정하는 것인지, App을 개발하는 것인지, Admin 기능을 확장하는 것인지** 구분할 필요가 있을 것 같다.

## 오늘 개선한 것

### 모바일 사이드바를 열었는데 배경이 함께 스크롤되던 문제

모바일 사이드바를 열었을 때 사이드바 뒤에 있는 본문까지 함께 스크롤되는 문제가 있었다.

사이드바가 열린 동안에는 사용자의 관심과 조작 대상이 사이드바에 있어야 한다. 하지만 배경까지 움직이면서 화면의 위치가 달라졌고, 사이드바를 닫은 뒤 원래 읽던 위치를 놓칠 수 있었다. 그래서 사이드바 토글 시 `document.body.style.overflow` 속성을 조작해 배경 스크롤을 막도록 했다.

사이드바 내부 구조도 함께 수정했다. 모바일이나 데스크톱으로 접근할 때 다크모드 토글 버튼 밑으로 컨텐츠가 보이는 문제가 있었다. Codex가 초기에 생성한 레이아웃을 그대로 사용하다 보니 예상치 못한 문제였다. 그래서 사이드바 내부 구조를 세로 방향 Flexbox 구조로 변경하고, 카테고리 영역만 남은 공간을 차지하며 스크롤되도록 만들었다.

이제 카테고리가 많아지더라도 카테고리 목록만 스크롤되고, 테마 전환 버튼은 사이드바 하단에 자연스럽게 유지된다. 단순히 `overflow` 속성 하나를 수정하는 문제처럼 보였지만, 실제로는 다음 두 스크롤 영역을 구분하는 것이 핵심이었다.

## 오늘의 결론

오늘은 알고리즘과 JavaScript 내장 객체를 공부하고, 새로운 업무에서 사용하게 될 Shopify를 살펴봤다. 그리고 TIL 사이트의 작은 UX 문제도 수정했다.

특히 오늘 작성한 코드들은 기능 자체보다 **왜 해당 자료구조와 속성을 선택했는지** 이해하는 과정이 중요했다.

- `myFlat()`에서는 재귀와 희소 배열의 동작을 이해했다.
- LRU Cache에서는 `Map`의 삽입 순서를 활용했다.
- Shopify에서는 플랫폼을 구성하는 각 영역의 역할을 구분했다.
- 사이드바에서는 페이지와 내부 컨테이너의 스크롤 책임을 분리했다.

예전에는 코드가 동작하면 빠르게 넘어가는 경우가 많았다. 하지만 이제는 구현한 코드가 왜 동작하는지, 실제 기능은 어떤 조건까지 고려해야 하는지 조금 더 꼼꼼하게 살펴보려고 한다.

오늘도 배운 것을 기록했고, 그 기록을 보여주는 공간을 한 단계 더 다듬었다.
